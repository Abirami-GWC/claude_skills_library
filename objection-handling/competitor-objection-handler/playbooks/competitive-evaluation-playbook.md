# Competitive evaluation playbook

Guidance for the evaluation stage of a deal specifically, when a prospect is actively benchmarking
against one or more competitors — broader than a single name-drop, which is what
`handlers/competitor-objection-handler.md` handles. Use this for call prep before an evaluation
conversation; use the handler for responding to a specific objection in the moment.

## Preparing for an evaluation-stage call

Confirm which competitors are actually in play before the call — from CRM notes, the prospect's
own statements, or a discovery call — so the right battlecard row is ready rather than reasoned
from scratch live. If a named competitor isn't in
`handlers/competitor-objection-handler.md`'s table yet, use
`templates/new-battlecard-template.md` to prepare one *before* the call rather than improvising an
unverified claim on the spot.

## Common evaluation-stage patterns

| Pattern | What's actually happening | Move |
|---|---|---|
| "We're doing a bake-off between a few vendors" | Prospect wants a structured comparison, not a sales pitch | Ask what criteria they're scoring on — this often reveals which differentiation angle will land, rather than guessing. |
| Prospect cites a feature the competitor has that this product doesn't | Feature-count comparison, which rarely favors a challenger on a raw list | Don't contest the feature gap directly if it's real — shift to the differentiation angle in the battlecard table (time-to-value, operational overhead) per Challenger-style reframing. |
| Prospect asks for references who switched from that competitor | Wants social proof, not just a claim | Only offer references that are real and confirmed available — don't imply availability to close the objection if none exists yet. |

## Keeping the battlecard current

Competitor facts go stale. Before a high-stakes evaluation call, it's worth confirming the
relevant battlecard row in `handlers/competitor-objection-handler.md` was verified recently —
if it's been a while, flag it for re-verification rather than relying on it as-is.
