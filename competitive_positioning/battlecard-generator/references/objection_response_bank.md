# Objection-Response Patterns for Battlecards

Patterns to adapt, not scripts to paste verbatim — always tailor to the actual
sourced facts for the competitor at hand.

**Canonical source note:** this file is the canonical version of these four
generic objection-scenario patterns. `why-us-messaging/references/objection_handling.md`
intentionally duplicates this same base content (so each skill stays
self-contained and independently installable) but adds persona-specific
framing on top. If these patterns change, update both copies and diff against
this file when in doubt about which is current. This is a different asset
from `objection_handling/competitor-objection-handler`'s named-competitor
table — that skill answers "what do I say when Fortinet/Palo Alto/SonicWall
gets named on a call," organized by competitor; this file answers "what's the
pattern for a cheaper/already-invested/feature-parity objection," organized
by scenario type, independent of which competitor raised it.

## "They said your product doesn't support X."

If true but nuanced (different approach, roadmap item): correct precisely and
calmly, citing the actual status. If simply false: correct factually without
editorializing about the competitor's honesty.

Pattern: "That's actually supported — [specific detail]. Happy to show you
live if that'd help."

## "Their pricing is cheaper."

Only reframe to total cost if there's real evidence (implementation cost,
overage fees, required add-ons). Otherwise pivot to value against the
prospect's stated priority.

Pattern: "On the sticker price, sure — worth comparing what's actually
included at that price versus what you'd need to add on to match [use case]."

## "We've already invested in their ecosystem / relationship."

Acknowledge switching cost as real; don't dismiss it. Offer a lower-risk next
step (pilot, side-by-side on one workflow).

Pattern: "Makes sense — switching cost is real. A lot of teams in your
position start with a side-by-side on just [specific workflow] before
deciding anything bigger."

## "Their support / SLA is better."

Only make a comparative claim if sourced. Otherwise state your own metric
without a false comparison.

Pattern: "Our median first-response time is [X] — happy to share how that's
measured. I don't have a verified number for theirs to compare directly."

## General principle

Every response should end by returning to the prospect's own stated priority
or a concrete next step, not by continuing to litigate the competitor
comparison indefinitely.
