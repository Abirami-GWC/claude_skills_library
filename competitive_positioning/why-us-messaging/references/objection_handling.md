# Objection-Handling Patterns

Adapt to the actual competitor/product facts and persona at hand — these are
patterns, not scripts.

**Canonical source note:** the four base patterns below are intentionally the
same ones maintained in
`battlecard-generator/references/objection_response_bank.md` — treat that
file as canonical and this copy as the persona-adapted variant (each skill
stays self-contained and independently installable, so the content is
duplicated rather than cross-referenced at runtime). If the underlying
objection logic changes, update both files. What this version adds on top:
explicit guidance on how the same objection should be framed differently per
buyer persona (see the pricing example below).

## "But [Competitor] told us they do that too."

Don't contradict flatly. Ask what specifically was described, then compare
precisely: "Worth checking whether that's native or an add-on / GA or
roadmap, since that affects [persona's stated priority]."

## "[Competitor] is cheaper."

Only reframe to total cost with real evidence. Otherwise pivot to value
against the persona's stated priority — this response differs sharply by
persona: a CFO wants a TCO breakdown; an end user doesn't care about pricing at all.

## "We've already invested in [Competitor]'s ecosystem."

Acknowledge switching cost as real. Offer a lower-risk next step (pilot,
side-by-side on one workflow) rather than pushing a full switch decision.

## "Your competitor said you don't support X."

If true but nuanced, correct precisely with a source. If false, correct
factually without editorializing about the competitor's honesty.

## General principle

Every objection response should end by returning to this persona's own stated
priority, not by continuing to litigate the comparison indefinitely.
